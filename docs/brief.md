# Project Brief: Claude Code Environment Manager (claudecm)

## Executive Summary

**Claudecm** 是一个专为 Claude Code 开发者设计的环境变量管理 CLI 工具，灵感来源于 kubecm 的成功设计模式。该工具解决了开发者在多个 Claude API 配置之间切换、管理认证令牌以及团队协作共享配置的核心痛点。

**核心价值主张：**
- 统一的多环境配置管理，支持无限配置文件存储
- 直观的交互式 TUI 界面，提供毫秒级环境切换体验
- 安全的加密存储机制，保护敏感 API 凭据
- 无缝的 shell 集成，支持 `eval $(claudecm export)` 一键加载

**目标市场：** Claude Code 个人开发者、DevOps 工程师、需要多环境配置的开发团队

**关键差异化优势：** 专为 Claude Code 生态量身定制，相比通用环境管理工具提供更精准的功能设计和更优的开发体验

## Problem Statement

### 当前状态与痛点

**Claude Code 开发者当前面临的核心挑战：**

1. **配置管理碎片化** - 开发者需要在不同项目、不同 API 提供商（Anthropic US/CN、Moonshot等）之间频繁切换，缺乏统一管理界面

2. **安全风险与便利性矛盾** - API 令牌通常以明文存储在 .env 文件中或硬编码在脚本里，既不安全也不便于管理

3. **团队协作效率低下** - 团队成员需要手动同步配置，容易出现配置不一致问题，缺乏标准化的配置共享机制

4. **开发工作流中断** - 切换环境需要手动编辑多个文件或运行复杂脚本，打断开发心流

### 问题影响量化

**生产力损失：** 每次环境切换平均耗时 2-5 分钟，按每日切换 3-5 次计算，每周浪费 30-60 分钟

**安全风险等级：** 高 - API 令牌泄露可能导致严重的财务和数据安全后果

**团队协作成本：** 中等 - 配置不一致导致的调试和沟通成本 estimated 15-20% 的开发时间

### 现有解决方案的不足

**通用工具的局限性：**
- **dotenv**: 仅支持静态配置，缺乏动态切换能力
- **direnv**: 基于目录的自动切换，不适合全局配置管理
- **自定义脚本**: 维护成本高，缺乏标准化界面和错误处理

**为什么现在必须解决这个问题：**
- Claude Code 采用率快速增长，市场需要专业化工具
- 远程开发和多环境部署成为新常态
- API 服务商地域化要求（如中国的合规要求）增加了配置复杂性

## Proposed Solution

### 核心概念与方法

**Claudecm** 采用**配置中心化管理 + 交互式切换**的解决思路，将所有 Claude Code 相关的环境变量组织成可命名的配置文件（profiles），通过直观的命令行界面提供即时切换能力。

**解决方案架构：**
- **存储层**: `~/.claudecm/` 目录下的 YAML 配置文件，支持加密存储
- **交互层**: 基于 survey/bubbletea 的 TUI 界面，提供直观选择体验
- **集成层**: 原生 shell 集成能力，支持 `eval $(claudecm export)` 无缝加载

### 关键差异化优势

**1. 专为 Claude Code 优化**
- 预设常用环境变量模板（ANTHROPIC_BASE_URL、ANTHROPIC_AUTH_TOKEN等）
- 智能验证配置有效性和兼容性
- 针对 Claude API 特殊需求的功能设计

**2. 用户体验至上**
- **毫秒级切换**: 本地配置文件读取，无网络延迟
- **记忆友好**: `claudecm switch` 提供历史和收藏功能
- **容错设计**: 配置验证、回滚机制、备份恢复

**3. 安全优先**
- **AES 加密存储**: 保护敏感 API 令牌
- **权限控制**: 严格的文件权限管理（600）
- **安全显示**: 令牌脱敏显示，防止意外泄露

### 为什么这个解决方案会成功

**学习成功经验**: 采用 kubecm 的成熟交互模式，用户学习成本低
- **开发者习惯契合**: 命令行工具优先，符合目标用户工作流
- **渐进式采用**: 可以从基础功能开始，逐步扩展到高级特性
- **生态集成**: 预留 IDE 插件、云同步等扩展接口

**产品愿景**: 成为 Claude Code 生态中不可或缺的基础设施工具，就像 kubectl 对于 Kubernetes、git 对于版本控制一样重要。

## MVP Scope

### 核心功能（必须具备）

**基础配置管理：**
- **claudecm add** - 交互式添加新配置，支持基本环境变量设置
- **claudecm list** - 显示所有可用配置及其基本信息
- **claudecm switch** - 交互式选择并激活配置（核心功能）
- **claudecm delete** - 删除指定配置

**导出与集成：**
- **claudecm export** - 输出当前激活配置的 shell export 语句
- **eval $(claudecm export)** 支持 - 一键加载环境变量

**存储基础：**
- 本地 YAML 配置文件存储（~/.claudecm/ 结构）
- 基本的配置验证和错误处理
- 当前激活配置状态管理

### MVP 范围外功能

**高级交互功能：**
- **claudecm rename** - 重命名配置（可以后续添加）
- **claudecm alias** - 配置别名系统（Nice-to-have）
- **claudecm merge** - 导入现有 .env/yaml 文件（高级功能）

**云与协作功能：**
- **claudecm cloud** - 云端同步（完整版本功能）
- **团队配置共享**
- **多设备同步**

**企业级功能：**
- 权限管理
- 审计日志
- SSO 集成

**扩展功能：**
- IDE 插件集成
- 多厂商支持（OpenAI、ZhipuAI 等）
- 配置模板系统

### MVP 成功标准

**核心价值验证：**
- 用户能够成功添加和管理多个 Claude Code 配置
- 环境切换时间 <10 秒（包括交互时间）
- 导出的环境变量能够正常工作
- 工具在三大平台（macOS、Linux、Windows）上稳定运行

**用户体验标准：**
- 新用户能在 5 分钟内完成首次配置设置
- 命令行界面直观易懂，无需频繁查阅文档
- 错误信息清晰，帮助用户快速定位问题

## Technical Considerations

### 平台需求

**目标平台：**
- **主要平台**: macOS、Linux（开发者主力平台）
- **次要平台**: Windows（企业用户支持）
- **架构支持**: x86_64、ARM64（Apple Silicon）

**性能要求：**
- **启动时间**: <100ms（本地操作响应时间）
- **内存占用**: <50MB（轻量级 CLI 工具）
- **存储占用**: <10MB（配置文件和程序本体）
- **并发支持**: 支持多个终端进程同时运行

### 技术偏好

**核心技术栈：**
- **编程语言**: Go 1.21+（推荐的 Go + Cobra）
- **CLI 框架**: github.com/spf13/cobra（命令行结构和解析）
- **交互组件**: github.com/AlecAivazis/survey/v2 或 charmbracelet/bubbletea
- **配置格式**: YAML（gopkg.in/yaml.v3）
- **加密库**: crypto/aes（敏感信息加密存储）

**构建与发布：**
- **构建工具**: Go 原生构建 + goreleaser（推荐的打包发布方案）
- **包管理**:
  - macOS: Homebrew
  - Linux:APT/YUM、Snap
  - Windows: Scoop、Chocolatey
- **CI/CD**: GitHub Actions（自动化测试和发布）

### 架构考虑

**代码结构：**
```
claudecm/
├── cmd/          # 命令行入口点
│   ├── root.go
│   ├── add.go
│   ├── switch.go
│   └── export.go
├── internal/     # 内部包
│   ├── config/   # 配置管理
│   ├── storage/  # 存储层
│   ├── crypto/   # 加密功能
│   └── ui/       # 交互界面
├── pkg/          # 公共包
└── scripts/      # 构建脚本
```

**存储架构：**
- **配置目录**: `~/.claudecm/`
- **权限管理**: 600（用户读写，其他用户无权限）
- **备份策略**: 自动备份修改前的配置
- **版本兼容**: 向后兼容的配置文件格式

### 安全与合规

**数据安全：**
- **传输加密**: 本地存储，无网络传输
- **存储加密**: AES-256 加密敏感字段
- **内存安全**: 及时清理内存中的敏感信息
- **日志安全**: 敏感信息不记录到日志文件

**权限控制：**
- **文件权限**: 严格的 Unix 权限设置
- **访问控制**: 仅限配置文件所有者访问
- **审计日志**: 可选的操作记录功能
