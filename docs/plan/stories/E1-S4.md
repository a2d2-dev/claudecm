# E1-S4 — `internal/storage/backup.go`: timestamped backup with verification

**User story.**
As a claudecm developer, I want a backup primitive that copies the original bytes of an owned file to `<basename>.bak.<ISO8601>.<short-uuid>` at `0600` under `~/.claudecm/backups/<tool>/`, verifies size after copy, and returns the backup path so the write-path can roll back from it.

**Acceptance criteria.**
- Given an owned file, when `backup.Create(toolID, filePath, bytes)` is called, then a file at `~/.claudecm/backups/<tool>/<basename>.bak.<ISO8601>.<uuid>` is created with mode `0600` and identical content.
- Given the backup size does not match the source size after copy, when verification runs, then `backup.Create` returns an error and the partial backup is removed.
- Given two backups created within the same second, when both are read back, then both files exist (UUID disambiguates).
- The backup contains the **pre-write bytes, untransformed** — no formatting, no key reordering.
- The backup directory is created with mode `0700` if missing.

**PRD/architecture refs.** PRD FR-5 step 6, FR-12; Architecture §4 step 6, §8.

**Test plan.**
- Happy: backup a 4 KB JSON blob → file exists at expected path, byte-identical, mode `0600`.
- Edge: two backups in the same second → both present; simulate size mismatch (mock writer truncates) → error returned, partial backup deleted.

**No fallback writes.** A failed backup is **fatal** for the calling write — `writepath.Apply` does not proceed to atomic write without a verified backup. FR-5.

**Complexity.** S.

**Dependencies.** E1-S2, E1-S3.

## Implementation Notes

**Decision — timestamp scheme.** Backup filenames use nanosecond-precision UTC timestamps (`<basename>.<YYYYMMDDTHHMMSSZnnnnnnnnn>`) and treat same-nanosecond collisions as `ErrTargetExists` rather than UUID-disambiguating. Deterministic, sortable, no random dependency; real-clock callers will never collide.
