# E6-S4 — `cmd/import {claude-code|codex}` with `--name`, `--yes`, `--overwrite`, `--dry-run`

**User story.**
As a Claude Code or Codex user, I want `claudecm import claude-code` or `claudecm import codex` to seed a named profile from my existing on-disk config, with a non-interactive `--name`/`--yes` mode for scripts.

**Acceptance criteria.**
- Given `claudecm import claude-code`, when it runs interactively, then the user reviews and names the imported profile before saving.
- Given `--name <name> --yes`, when `import` runs, then it saves the imported profile non-interactively at the given name.
- Given `--name` collides with an existing profile, when `import` runs, then it refuses unless `--overwrite` is also passed.
- Given `--dry-run`, when `import` runs, then it prints what the profile would be without writing.
- Imported profiles carry `schema_version: 1`.
- Round-trip fidelity ≥ 95% on the fixture corpus (SM-2).

**PRD/architecture refs.** PRD FR-8, FR-15, SM-2, NFR-M1; Architecture §3, §7.

**Test plan.**
- Happy: `import claude-code --name official --yes` → profile created, round-trip byte-identical on owned-key scope.
- Edge: `--name` collision without `--overwrite` → refused; malformed on-disk file → refused (NFR-S1); `--dry-run` → no write.

**No fallback writes.** Malformed source → refuse, no profile candidate, no write. NFR-S1.

**Complexity.** M.

**Dependencies.** E3-S3, E4-S3, E6-S3.
