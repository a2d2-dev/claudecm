# E1-S5 — `internal/storage/retention.go`: N=10 pruning + `~/.claudecm/audit.log`

**User story.**
As a claudecm developer, I want bounded backup retention with an audit log so that `~/.claudecm/backups/<tool>/` does not grow unbounded, every pruned backup is recorded, and `restore` is never blocked by retention.

**Acceptance criteria.**
- Given N+1 backups for a `(tool, file)` tuple, when retention runs after a successful write, then the oldest is pruned, leaving N (default 10).
- Given `--retention <int>` is set on the invocation, when retention runs, then the override is used in place of the default.
- Given a prune occurs, when the audit log is read, then `~/.claudecm/audit.log` (mode `0600`) has a line per deleted backup with timestamp + path.
- Given a file in the backup dir that does not match `<owned-file-basename>.bak.*`, when retention runs, then it is **not** touched.
- Given the just-written target, when retention runs, then the target is **never** pruned (R2).
- `restore --list` shows whatever backups are present regardless of retention (R3).

**PRD/architecture refs.** PRD NFR-R1, NFR-R2, NFR-R3; Architecture §4 (retention).

**Test plan.**
- Happy: write 11 backups → oldest pruned, 10 remain, audit-log line added.
- Edge: put an unrelated file in `backups/codex/` named `notabackup.txt` → retention leaves it; set `--retention 3` and write 4 backups → 3 remain; verify audit-log mode `0600`.

**No fallback writes.** Retention failure does not roll back the user's successful write; it surfaces an error but the write stays committed. Refuse-on-malformed audit log is fine (recreate cleanly).
