# E4-S3 — `codex`: `Import` reads `config.toml` + `auth.json`

**User story.**
As a Codex CLI user, I want `claudecm import codex` to seed a profile from my working `~/.codex/config.toml` and `~/.codex/auth.json` so I can adopt claudecm without retyping credentials or provider config.

**Acceptance criteria.**
- Given valid `config.toml` + `auth.json`, when `codex.Import` runs, then it returns `CoreFromTool` (base_url, model, model_provider, api_key from auth.json) and `OverlayFromTool` (per-provider deviations stored in `Raw`).
- Given `auth.json` present but `config.toml` missing (or vice versa), when `Import` runs, then it returns what it can with a clear note in the report (importing one half is allowed).
- Given an unparseable file, when `Import` runs, then it returns an error and produces no profile candidate (NFR-S1).
- Round-trip `Import → Render → Apply` byte-identical on owned-key scope for each fixture.

**PRD/architecture refs.** PRD FR-8, §4.7, NFR-S1, NFR-T1; Architecture §3, §3.1.

**Test plan.**
- Happy: minimal `config.toml` + `auth.json` with `OPENAI_API_KEY` → round-trips byte-identical on owned-key scope.
- Edge: malformed TOML → refuse; `auth.json` missing → import config-only with a clear "no credentials imported" line.

**No fallback writes.** Malformed import refused. NFR-S1.

**Complexity.** M.

**Dependencies.** E4-S1, E4-S2.

## Implementation Notes / Decisions

The AC #2 phrase "with a clear note in the report" was NOT delivered as a `Notes` field on the adapter `Import` signature. It is delivered one layer up, in the CLI at `cmd/import` (see E6-S4), by composing the adapter's `Presence.Files` output with the `Import` result to render user-facing partial-import diagnostics ("no credentials imported", "config-only import", etc.).

Rationale for keeping the adapter surface as `(CoreFromTool, OverlayFromTool, error)`:

- The adapter contract in `internal/adapter/adapter.go` locks the same `(Core, Overlay, error)` shape across every tool. Sneaking a codex-specific `Notes` slot in would either fork the interface or force every other adapter (claudecode, and future tools) to grow a field they cannot meaningfully populate.
- Presence + import outcome are already independently exposed: `Adapter.Presence(ctx, r)` reports per-file presence in the `PresenceFile` shape defined by E4-S1; the CLI composes that with `Import`'s zero-value Core/Overlay signal to distinguish "nothing to import" from "config-only" from "auth-only" from "both partial-but-owned-empty" and print the right message.
- Downstream tests can pin the CLI-level report shape without coupling to adapter internals.

The AC's implied return-shape ("a clear note in the report") therefore supersedes at the CLI layer; the adapter's job stays "return the extracted candidate and refuse-on-malformed", per NFR-S1.
