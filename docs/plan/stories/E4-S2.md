# E4-S2 — TOML parser plumbing via `pelletier/go-toml/v2` document model

**User story.**
As a claudecm developer, I want a thin TOML doc-model wrapper inside the Codex adapter that round-trips arbitrary `config.toml` files preserving comments + key order (NFR-S7), so non-owned keys survive and "comments/order may shift" warnings fire only when truly necessary.

**Acceptance criteria.**
- Given any well-formed `config.toml` (with comments above keys, inline comments, and section ordering), when load + round-trip through the wrapper, then output bytes are byte-identical to input.
- Given a write that updates an owned key, when emitted, then comments + key order around non-owned keys are preserved; any unavoidable reordering surfaces a `comments/order may shift` warning (NFR-S7).
- Stdlib `encoding/json` and unsafe TOML libs are not used; only `pelletier/go-toml/v2` doc model.
- The wrapper is unit-tested independently of the Codex adapter's owned-key logic.

**PRD/architecture refs.** PRD NFR-S7; Architecture §10.

**Test plan.**
- Happy: load + dump the canonical maximal `config.toml` fixture → byte-identical.
- Edge: load a `config.toml` with `[model_providers.X]` followed by a comment block → set `model_providers.X.base_url` → comment preserved; if a re-order is forced, warning emitted.

**No fallback writes.** Parse failure refused; no "best-effort" TOML rewrite. NFR-S1.

**Complexity.** M.

**Dependencies.** E4-S1.
