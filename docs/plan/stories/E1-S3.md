# E1-S3 — `internal/storage/atomic.go`: temp+fsync+rename, `O_CREAT|O_EXCL` first-write

**User story.**
As a claudecm developer, I want a single atomic-write primitive so that every write inside `writepath.Apply` (and profile/state writes) follow the same temp-file → fsync → rename ritual and the first write against a missing target uses `O_CREAT|O_EXCL` semantics.

**Acceptance criteria.**
- Given a target path on the same filesystem, when `atomic.WriteFile(target, bytes, mode)` is called, then it writes to `<target>.tmp.<pid>.<rand>`, fsyncs, and renames over `target`; mode `0600`.
- Given the target file does not yet exist, when `atomic.WriteFile` is called, then the temp file is opened with `O_CREAT|O_EXCL` and a `rename` is performed against the missing target.
- Given the target appears between read and rename, when the rename is attempted, then a concurrent-edit error is returned and the temp file is removed.
- Given a write that fails after the temp file is created (disk full, fsync error), when the function returns, then the temp file is unlinked and the original target is untouched.
- Mode is re-asserted on every write; the helper does not assume the file exists with the right mode.

**PRD/architecture refs.** PRD FR-5 step 7, NFR-C3; Architecture §4 step 7, §8.

**Test plan.**
- Happy: write 1 KB and 1 MB payloads; verify content + mode `0600` after rename; verify temp file is gone.
- Edge: pre-create target between `WriteFile` start and end → concurrent-edit error; force fsync error via a mock fs → original file intact, temp removed.

**No fallback writes.** A failed write **does not** truncate or partially overwrite the target; no "best-effort" recovery. NFR-S1.

**Complexity.** M.

**Dependencies.** E1-S2.
