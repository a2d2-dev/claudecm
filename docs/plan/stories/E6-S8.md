# E6-S8 — `cmd/export` refactor (secondary path; `--format yaml`, `--redact`)

**User story.**
As a claudecm user, I want `claudecm export` to emit shell `export VAR=...` lines suitable for `eval $(claudecm export)`, plus an optional YAML output and a `--redact` flag for capture-to-log workflows.

**Acceptance criteria.**
- Given an active (or named) profile, when `export` runs with no flags, then it emits `export VAR=...` lines covering each tool's env-var contract (per the EnvOverride allowlist) in **plaintext** because the command is designed for `eval $(claudecm export)` (NFR-S8 carve-out).
- Flag surface (must be exhaustively wired in `cmd/export`):
  - `--format <shell|yaml>` (default `shell`): `shell` emits `export VAR=...` lines; `yaml` emits the profile in profile-native YAML form (round-trippable with `import`).
  - `--redact` (default `false`): when set, secrets are redacted as `sk-***last4` in both `shell` and `yaml` output. When unset, output is plaintext regardless of `--format`.
  - Mixing `--format yaml --redact` redacts the YAML document's `api_key` field; mixing `--format shell --redact` redacts the `VAR=...` values.
- Output is consistent with `current` and `explain` for the same profile (same effective values).
- `export` never writes to tool files; it is the documented secondary activation path.

**PRD/architecture refs.** PRD FR-9, NFR-S8; Architecture §7.

**Test plan.**
- Happy: `export` of active profile → expected `export ANTHROPIC_API_KEY=... ANTHROPIC_BASE_URL=...` lines; `--format yaml` round-trips.
- Edge: `--redact` redacts; mixing `--format yaml --redact` redacts the YAML's `api_key`.

**No fallback writes.** Read-only.

**Complexity.** S.

**Dependencies.** E5-S2.
