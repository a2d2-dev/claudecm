# E4-S5 — `codex`: `Apply` (per file, routed via writepath; cross-file ordering deferred to E7)

**User story.**
As a claudecm developer, I want `codex.Apply` to forward each per-file `WritePlan` to `internal/writepath.Apply` — never directly opening files for write — so the FR-5 invariant covers both Codex files identically.

**Acceptance criteria.**
- Given a list of per-file plans, when `codex.Apply(plans)` is called, then it iterates and forwards each to `writepath.Apply`, returning per-file `ApplyReport`s + warnings.
- The adapter never calls `os.WriteFile` / `os.Rename` / `os.Create` on owned files. CI lint rule guards this.
- Cross-file ordering and rollback are **not** this story's job — the commit package (E7) sequences and ordered-renames; this story produces the per-file pieces.
- On `writepath.Apply` error, the adapter wraps with `codex` context.

**PRD/architecture refs.** PRD FR-5, FR-16; Architecture §3, §4, §5; coding-standards rules 1, 13.

**Test plan.**
- Happy: single-file plan for `config.toml` writes through writepath → backup created, file updated.
- Edge: writepath returns `ErrConcurrentEdit` for `auth.json` → adapter surfaces wrapped error; lint check fails if any direct `os.Rename` is added.

**No fallback writes.** Adapter delegates. FR-5.

**Complexity.** S.

**Dependencies.** E2-S2 + E2-S3 + E2-S4, E4-S4.
