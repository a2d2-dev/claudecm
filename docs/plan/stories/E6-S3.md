# E6-S3 — `cmd/add` refactor to unified schema + `--dry-run`

**User story.**
As a claudecm user, I want `claudecm add` to create a profile in the unified schema (core + optional sparse `tools` overlay) with `schema_version: 1`, supporting `--dry-run` and refusing to silently overwrite an existing name.

**Acceptance criteria.**
- Given a profile name + core flags (`--base-url`, `--api-key`, `--model`, etc.), when `add` runs, then it writes `~/.claudecm/profiles/<name>.yaml` with `schema_version: 1` and timestamps.
- Given an existing name, when `add` runs without `--overwrite`, then it refuses with a clear error.
- Given `--set tools.claude_code.model=...` style flags, when `add` runs, then it populates the sparse overlay map.
- Given `--dry-run`, when `add` runs, then it prints the would-be profile and exits 0 without writing.
- Given the name fails the profile-name regex (`^[a-z0-9][a-z0-9._-]{0,63}$`, NFR-S5), when `add` runs, then it refuses with a clear error.
- `add` does NOT automatically activate the new profile; activation is a `switch` step.

**PRD/architecture refs.** PRD FR-1, FR-15, NFR-S5, NFR-M1; Architecture §2.1, §7.

**Test plan.**
- Happy: `claudecm add work --base-url ... --api-key ...` → profile file written; `list` shows it.
- Edge: name `../evil` → refused; existing name → refused without `--overwrite`; `--dry-run` → printed only.

**No fallback writes.** Invalid name / existing name / bad schema → refuse, no write.

**Complexity.** M.

**Dependencies.** E1-S1, E1-S2, E1-S7.
