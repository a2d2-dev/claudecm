# E5-S3 — EnvOverride per-tool allowlist + `envextract` wiring

**User story.**
As a claudecm developer, I want the EnvOverride layer to consider **only** the per-tool allowlist from NFR-E1, so `explain` never falsely claims an unrelated env var is shadowing the on-disk config.

**Acceptance criteria.**
- Claude Code EnvOverride allowlist (verbatim): `ANTHROPIC_API_KEY`, `ANTHROPIC_AUTH_TOKEN`, `ANTHROPIC_BASE_URL`, `ANTHROPIC_MODEL`, `ANTHROPIC_SMALL_FAST_MODEL`, `CLAUDE_CODE_USE_BEDROCK`, `CLAUDE_CODE_USE_VERTEX`.
- Codex EnvOverride allowlist (verbatim): `OPENAI_API_KEY`, `OPENAI_BASE_URL`, `CODEX_HOME`, `CODEX_MODEL`, `CODEX_MODEL_PROVIDER`.
- Allowlists are exported `var`s in their respective adapter packages; CI fixture matrix exercises them.
- `internal/envextract` is the only source of env reads for the resolver.
- `explain --all-env` may surface other env vars for diagnostic display, but they NEVER shadow a layer.

**PRD/architecture refs.** PRD NFR-E1; Architecture §6.1.

**Test plan.**
- Happy: setting `ANTHROPIC_API_KEY` shows EnvOverride wins for Claude Code only.
- Edge: setting `RANDOM_ENV` does not shadow any layer; `explain --all-env` lists it under a diagnostic section without altering winning layers.

**No fallback writes.** N/A — read-only.
