# E2-S4 — Concurrent-edit detection (size + mtime + sha256, exit code 2)

**User story.**
As a claudecm developer, I want `writepath.Apply` to record `(size, mtime, sha256)` of the target at step 2 and re-check before step 7 rename, so concurrent external edits are detected and aborted cleanly with the backup retained.

**Acceptance criteria.**
- Given step 2 records `(size, mtime, sha256)`, when step 7 is about to rename, then re-stat + re-hash and compare; any mismatch → abort.
- Given an abort, when `Apply` returns, then the backup created at step 6 is **retained**, the error is `ErrConcurrentEdit` with the backup path, and the CLI exit code is 2.
- Given a first-write against a missing target (NFR-C3), when between steps 2 and 7 the target appears, then `Apply` aborts as concurrent-edit conflict (no overwrite of the new file).
- The temp file is removed on abort; the target is untouched.

**PRD/architecture refs.** PRD NFR-C2, NFR-C3, FR-5 step 9; Architecture §4 step 9; coding-standards rule 6.

**Test plan.**
- Happy: no external edit between read and rename → write succeeds.
- Edge: between read and rename, externally modify the target (size + content change) → abort, backup retained, exit code 2; first-write race: target appears after step 2 → abort.

**No fallback writes.** Concurrent edit aborts; no "merge our write into their change". NFR-C2.

**Complexity.** M.

**Dependencies.** E2-S2.
