# E3-S3 — `claudecode`: `Import` reads `~/.claude/settings.json` into Core + Overlay

**User story.**
As a Claude Code user, I want `claudecm import claude-code` to seed a profile from my working `~/.claude/settings.json` so I can adopt claudecm without retyping anything.

**Acceptance criteria.**
- Given `~/.claude/settings.json` with valid JSON/JSONC, when `claudecode.Import` runs, then the adapter returns `CoreFromTool` (base_url, api_key/auth_token, model, small_fast_model) and `OverlayFromTool` (tool-specific deviations, including `CLAUDE_CODE_USE_BEDROCK` / `CLAUDE_CODE_USE_VERTEX`).
- Given an unparseable settings.json, when `Import` runs, then it returns a clear error and produces no profile candidate (NFR-S1).
- Given a settings.json with non-owned keys (`permissions`, `hooks`, `mcpServers`, `theme`), when `Import` runs, then those keys appear in `OverlayFromTool.Raw` so the round-trip can re-emit them byte-for-byte, but they do not appear in the Core.
- Round-trip on the fixture corpus: `Import → Render → Apply` produces a settings.json byte-identical on owned-key scope (SM-2 helper).

**PRD/architecture refs.** PRD FR-8, §4.7, NFR-S1, NFR-T1; Architecture §3, §3.1.

**Test plan.**
- Happy: minimal canonical settings.json + maximal canonical settings.json → round-trips byte-identical.
- Edge: malformed JSON → refuse; unknown keys (`mcpServers`, `theme`) → preserved in `Raw` and surfaced as "preserved verbatim" on `switch`.

**No fallback writes.** Malformed import refused. NFR-S1.

**Complexity.** M.

**Dependencies.** E3-S2.

## Implementation Notes / Decision

Non-owned keys (`permissions`, `hooks`, `mcpServers`, `theme`, and any
future unowned knob) are NOT copied into `OverlayFromTool.Raw` during
`Import`. The round-trip guarantee is delivered by FR-5 merge-preserve
at `Apply` time: `Apply` reads the current on-disk `settings.json`,
replaces owned keys with the Profile's values, and preserves everything
else byte-for-byte. Double-sourcing (storing unowned keys in BOTH the
file AND `Overlay.Raw`) would stale-fork on user hand-edits between
`Import` and the next `Apply`: the on-disk file would carry the fresh
hand-edit while `Overlay.Raw` would still hold the stale snapshot
captured at `Import` time, and the merge-preserve rule would have no
way to decide which side wins without re-adopting the on-disk value.

This decision supersedes the original phrasing of AC #3 (which said
those keys "appear in `OverlayFromTool.Raw`"). The observable AC —
byte-identical round-trip on the owned-key scope, unowned keys
preserved verbatim on switch — still holds; only the mechanism has
moved from `Raw` to merge-preserve at `Apply` time.
