# E1-S6 — `internal/storage/lock.go`: `gofrs/flock` wrapper + `--lock-timeout`

**User story.**
As a claudecm developer, I want a single advisory-lock wrapper so that every owned-file write acquires `flock(LOCK_EX)` with a 5-second default timeout (override via `--lock-timeout`) and the lock is always released, including on error.

**Acceptance criteria.**
- Given a target path, when `lock.AcquireExclusive(path, timeout)` is called and no other process holds the lock, then it returns a `Lock` handle in <100 ms on a warm machine.
- Given another process already holds the lock, when `AcquireExclusive` is called with a 5 s timeout, then it returns `ErrLockBusy` after the timeout with the path in the error.
- Given the lock holder panics or errors mid-write, when `defer lk.Release()` runs, then the lock is released.
- Given `--lock-timeout 30s` is set, when `AcquireExclusive` is called, then the wait is up to 30 s.
- Lock acquisition order for multi-file writes (E7): `auth.json → config.toml → settings.json`; release is reverse.

**PRD/architecture refs.** PRD NFR-C1; Architecture §4 step 1, §5.

**Test plan.**
- Happy: acquire + release; verify second call after release succeeds; verify mode is honored.
- Edge: hold the lock in a goroutine and verify timeout returns `ErrLockBusy`; force panic in the lock holder and verify release fires.

**No fallback writes.** Lock timeout aborts the write outright; no "best-effort" lockless write, ever. NFR-C1.

**Complexity.** S.

**Dependencies.** E1-S2.
