# E7-S3 — `commit.Commit` (phase 2: ordered rename + post-write reparse)

**User story.**
As a claudecm developer, I want phase 2 of the two-phase commit to perform `rename` in the locked order (`auth.json → config.toml → settings.json`) with post-write reparse after each rename, so a failure on a later file can roll back already-committed earlier files.

**Acceptance criteria.**
- Given a `StagedTxn`, when `Commit` runs, then for each per-file plan it:
  1. Re-checks the FR-5 concurrency fingerprint (NFR-C2).
  2. Renames `<file>.tmp.<...>` over `<file>`.
  3. Post-write reparses; if it fails, triggers rollback (E7-S4).
- Locks are released in reverse order after the whole commit completes or rolls back (NFR-C1).
- Order of rename is exactly: `~/.codex/auth.json` first, then `~/.codex/config.toml`, then `~/.claude/settings.json`. So credentials commit before downstream config (architecture §5).
- On full success, `CommitReport.PerFile` reports `committed` for each file with backup path + post-fingerprint.

**PRD/architecture refs.** PRD FR-5, FR-16, NFR-C1, NFR-C2; Architecture §5.

**Test plan.**
- Happy: stage + commit two plans → both files updated in order, locks released reverse.
- Edge: concurrent edit detected mid-commit on second file → abort + rollback first file from backup (E7-S4 covers rollback assertion).

**No fallback writes.** Reparse failure → rollback, not "best-effort fix". FR-5.

**Complexity.** M.

**Dependencies.** E7-S2.
