# E3-S4 — `claudecode`: `Plan` + render via `sjson`/`gjson`

**User story.**
As a claudecm developer, I want the Claude Code adapter `Plan(profile)` function to produce a `WritePlan` whose intended bytes are the existing settings.json with **only the owned keys updated** via `sjson`, so comments, key order, indentation, and non-owned keys are preserved exactly.

**Acceptance criteria.**
- Given an existing settings.json (with comments, weird indentation, and unknown keys), when `Plan(profile)` runs, then the intended bytes equal the original bytes with only the owned keys re-written via `sjson.SetBytes`.
- Given the profile omits an owned key, when `Plan` runs, then under overlay-as-truth (NFR-S6) the owned key is removed from the intended bytes (so the tool falls back to its default).
- Given comment / order preservation is impossible for an owned key, when `Plan` runs, then the resulting `WritePlan.Warnings` includes `"comments/order may shift in <file>"` for the affected key. NFR-S7.
- `Plan` is pure: no disk writes; no syscalls beyond reading the current target.
- The owned-key diff in `WritePlan.Diff` powers `--dry-run` and `switch`'s pre-apply diff.

**PRD/architecture refs.** PRD FR-4 (pre-apply diff), FR-5, NFR-S6, NFR-S7; Architecture §3, §10.

**Test plan.**
- Happy: settings.json with comments + non-owned keys → render preserves them byte-for-byte; only `env.ANTHROPIC_*` keys change.
- Edge: profile omits `ANTHROPIC_MODEL` → owned key removed (overlay-as-truth); existing key with non-string type → render returns an error rather than coercing (no fallback).

**No fallback writes.** Type mismatch / parser limitation → refuse + warn loudly; never silently rewrite. NFR-S1 / NFR-S7.

**Complexity.** L.

**Dependencies.** E3-S2, E3-S3.
