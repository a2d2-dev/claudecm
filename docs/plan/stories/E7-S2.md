# E7-S2 — `commit.Stage` (phase 1: FR-5 steps 1–7 with deferred rename)

**User story.**
As a claudecm developer, I want phase 1 of the two-phase commit to perform FR-5 steps 1–7 (lock → read → parse → resolve symlink → diff → backup → atomic temp write) **without** the final rename, so phase 2 can rename in order with rollback semantics.

**Acceptance criteria.**
- Given `[]WritePlan` for multiple owned files, when `Stage(plans)` runs, then for each plan it acquires the lock (ordered: `auth.json → config.toml → settings.json`), reads + parses + diffs + backs up, and writes `<file>.tmp.<pid>.<rand>` — but does **not** rename.
- The `StagedTxn` carries per-plan `(lock, backupPath, tempPath, fingerprint)`.
- Any failure during phase 1 releases all locks (reverse order) and removes any temp files written so far.
- Lock acquisition order is encoded in code (NFR-C1).

**PRD/architecture refs.** PRD FR-5 steps 1–7, FR-16, NFR-C1; Architecture §4, §5.

**Test plan.**
- Happy: stage two plans → two backups + two temp files exist; no rename has happened.
- Edge: parse failure on second file → first file's temp is removed, both locks released, backup of first file may exist on disk but is not promoted.

**No fallback writes.** Parse failure refused. NFR-S1.

**Complexity.** M.

**Dependencies.** E2-S2, E2-S3, E2-S4, E7-S1.
