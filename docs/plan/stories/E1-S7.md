# E1-S7 — `~/.claudecm/` layout: `profiles/`, `state.yaml`, `backups/<tool>/`, `audit.log`

**User story.**
As a claudecm user, I want a predictable on-disk layout under `~/.claudecm/` so that profiles, active-profile state, backups, and the audit log all live in known places at the right modes.

**Acceptance criteria.**
- Given a fresh install, when `internal/storage` is first invoked, then `~/.claudecm/` is created mode `0700` with subdirs `profiles/`, `backups/`, and an empty `state.yaml` (schema_version 1, active_profile empty).
- Given the existing legacy layout (current `~/.claude/claudecm/` or similar from the brownfield code), when first invoked, then storage either reads the new path or returns a clear "run `claudecm` to initialize" error — no silent migration of unknown content.
- Profiles live at `~/.claudecm/profiles/<name>.yaml`, mode `0600`.
- State lives at `~/.claudecm/state.yaml`, mode `0600`, schema includes `ActiveProfile`, `LastSwitched`, `LastAppliedPerTool` per Architecture §2.2.
- Backups live at `~/.claudecm/backups/<tool>/<basename>.bak.*`.
- Audit log lives at `~/.claudecm/audit.log`, mode `0600`.
- Every read of an existing file re-asserts mode (no silent re-permission grants).

**PRD/architecture refs.** PRD §4.1 FR-1, NFR-S4; Architecture §2.2, §8.

**Test plan.**
- Happy: bootstrap on a fresh `--home /tmp/x` → all dirs/files present at correct modes.
- Edge: pre-create `~/.claudecm/state.yaml` with wrong mode `0644` → first read re-asserts to `0600`; pre-create with `schema_version: 2` → refuse.

**No fallback writes.** State YAML with bad schema is **refused**; no auto-migration. NFR-M1.

**Complexity.** M.
