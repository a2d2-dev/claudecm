# E5-S2 — `resolver.Resolve` layered-chain implementation

**User story.**
As a claudecm developer, I want one `Resolve(profile, tool)` function that walks the locked layer chain — built-in default → profile core → profile overlay → on-disk tool config → EnvOverride — and reports the winning + shadowed layers per field, so `current` and `explain` reuse one truth.

**Acceptance criteria.**
- Given a profile + tool, when `Resolve` runs, then it produces an `EffectiveView` with one `EffectiveField` per owned key and one per declared core field.
- Layer order is **lowest precedence first**: `BuiltInDefault < ProfileCore < ProfileOverlay < OnDiskToolConfig < EnvOverride` (PRD FR-7).
- For each field, the `WinningLayer` is the highest-precedence layer with a non-empty value.
- For each field, `ShadowedLayers` lists every lower layer that had a value, in precedence order.
- A key absent from every layer falls to `BuiltInDefault` and is reported as such.
- 100% correctness on the `explain` fixture matrix (SM-4).

**PRD/architecture refs.** PRD FR-7, SM-4; Architecture §6.

**Test plan.**
- Happy: profile core only → core wins; profile overlay set → overlay wins, core shadowed.
- Edge: env var set → env wins, on-disk shadowed; on-disk set + env unset → on-disk wins; field absent in all layers → built-in default reported.
