# E1-S2 — `internal/storage/paths.go`: HOME sanity, `--home`, profile-name regex, outside-`$HOME` refuse

**User story.**
As a claudecm developer, I want a single legal path-construction module so that every absolute path used by the rest of the project is built the same way, profile names cannot traverse out of the profile dir, and `$HOME` overrides are validated up front.

**Acceptance criteria.**
- Given an unset `$HOME` and no `--home`, when any path is requested, then `paths` returns a clear refuse-with-error and the process never proceeds with a broken HOME.
- Given `--home /tmp/sandbox` pointing at an absolute path owned by the current user, when paths are requested, then they resolve under `/tmp/sandbox/.claudecm/...`.
- Given a profile name `../evil` or `/etc/passwd` or `..`, when `ValidateProfileName` is called, then it returns an error; the regex used is `^[a-z0-9][a-z0-9._-]{0,63}$` (architecture §8).
- Given a tool-config target path that resolves (after symlink follow) outside `$HOME`, when paths.ResolveOwnedTarget is called, then it refuses with the resolved path in the error (NFR-S2).
- `filepath.Join` over user input outside this package is a coding-standards violation; this package is the only export of expansion + resolve helpers.

**PRD/architecture refs.** PRD NFR-S2, NFR-S3, NFR-S5; Architecture §8.

**Test plan.**
- Happy: `~/.claudecm/profiles/foo.yaml` resolves under `$HOME`; `--home /tmp/x` redirects all paths under `/tmp/x/.claudecm/...`.
- Edge: `..`, `/`, `\`, empty, 65-char name → `ValidateProfileName` errors; symlink at `~/.claude/settings.json` pointing to `/etc/passwd` → `ResolveOwnedTarget` refuses.

**No fallback writes.** Bad `$HOME` and bad profile names are **refused**; no auto-fix, no path coercion. NFR-S3 / NFR-S5.
