# E3-S1 — `Adapter` interface contract + `internal/adapter/` skeleton

**User story.**
As a claudecm developer, I want the `Adapter` interface (Detect, Files, Import, Plan, Apply, Project) declared in `internal/adapter` so both adapters and the commit/resolver packages compile against one contract.

**Acceptance criteria.**
- `internal/adapter/adapter.go` declares `Adapter`, `Presence`, `OwnedFile`, `Files`, `CoreFromTool`, `OverlayFromTool`, `WritePlan` (re-export from `writepath`), `ApplyReport`, `EffectiveView`.
- `internal/adapter/claudecode/` and `internal/adapter/codex/` exist as package skeletons; no business logic.
- `ToolID` is a typed string constrained to `"claude_code"` and `"codex"` in v1 (architecture §2.1).
- Adding a third tool (post-v1) is documented as a new package + commit-order registration only — no changes to writepath / commit / resolver.
- Doc comment on `Adapter` cites PRD §4.7 and architecture §3.

**PRD/architecture refs.** PRD §4.7; Architecture §3, §3.1.

**Test plan.**
- Happy: a synthetic adapter compiles against the interface and is usable in `writepath` tests.
- Edge: any unknown `ToolID` constructor returns an error.

**No fallback writes.** N/A — interface only.

**Complexity.** S.

**Dependencies.** E2-S1.
