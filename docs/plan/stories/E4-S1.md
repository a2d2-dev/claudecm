# E4-S1 — `codex`: `Detect` + `Files` + per-file owned-key allowlist `var`s

**User story.**
As a claudecm developer, I want the Codex adapter to own exactly two files (`~/.codex/config.toml`, `~/.codex/auth.json`), publish per-file owned-key allowlists as exported Go `var`s, and detect Codex installs from those file paths.

**Acceptance criteria.**
- `codex.Detect()` returns installed when either file exists; missing both → not installed.
- `codex.Files()` returns two `OwnedFile` entries with correct formats (`TOML`, `JSON`).
- `codex.OwnedKeysConfigTOML` is an exported `var []string` of concrete flat keys matching writepath.Flatten output — the frozen v1 set is enumerated in PRD §4.7 and covers the top-level knobs plus the explicit `openai` and `anthropic` provider entries.
- `codex.OwnedKeysAuthJSON` is an exported `var []string` enumerating the concrete flat leaves of the OAuth token bundle plus `OPENAI_API_KEY` per PRD §4.7.
- Custom provider names outside the v1 frozen set (e.g. `my-relay`) are considered non-owned by claudecm and preserved via FR-5 merge-preserve. Dynamic provider ownership is post-v1.
- Adding a key requires a paired PRD §4.7 edit (coding-standards rule 4).

**PRD/architecture refs.** PRD §4.7 (Codex); Architecture §3.1; coding-standards rule 4.

**Test plan.**
- Happy: both files present → detect installed; allowlists round-trip through `Files()`.
- Edge: only `auth.json` present → still installed; a `config.toml` with a custom provider (e.g. `relay-a`) has that provider's keys preserved verbatim via FR-5 merge-preserve rather than owned.

**No fallback writes.** N/A — declarations only.
