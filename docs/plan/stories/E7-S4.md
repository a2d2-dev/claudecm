# E7-S4 — Rollback: rename FR-5 step-6 backups over already-committed targets

**User story.**
As a claudecm developer, I want phase-2 rollback to restore every already-committed target from its phase-1 backup so a failed multi-file switch never leaves the user with a half-switched state.

**Acceptance criteria.**
- Given a phase-2 failure on file N (rename fail OR post-write reparse fail), when rollback runs, then for each file 1..N-1 already committed, the FR-5 step-6 backup is renamed over the target.
- Given the rollback rename itself fails on some file, when `Commit` returns, then `PartialFailure` lists that file's status as `rollback-failed` with both errors; exit code 3.
- After successful rollback, each rolled-back file is byte-identical to its pre-commit content.
- Locks are released only after rollback completes (NFR-C1 reverse order).

**PRD/architecture refs.** PRD FR-16; Architecture §5.

**Test plan.**
- Happy (failure path): commit `auth.json` succeeds → reparse fails on `config.toml` → `auth.json` rolled back byte-identical; both locks released.
- Edge: rollback rename on `auth.json` fails (read-only fs) → `PartialFailure` reports `rollback-failed`, exit code 3, both errors surfaced.

**No fallback writes.** Rollback is rename-only; never a "best-effort merge". FR-16.

**Complexity.** M.

**Dependencies.** E7-S3.
