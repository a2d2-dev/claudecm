# E2-S5 — Synthetic-adapter unit test matrix (≥ 80% coverage on `internal/writepath`)

**User story.**
As a claudecm developer, I want a synthetic in-memory adapter used solely by `internal/writepath` tests so the pipeline can be exercised end-to-end without dragging in real Claude Code / Codex parsing — and so coverage on `writepath` hits the NFR-T1 gate.

**Acceptance criteria.**
- Given a synthetic adapter implementing the `Adapter` interface with controllable failure injection (parse fail / rename fail / reparse-drift / concurrent-edit), when the test matrix runs, then every FR-5 branch is hit.
- Coverage on `internal/writepath` is **≥ 80%** line coverage.
- Each branch is a table-driven row: happy / malformed / symlink-out-of-home / concurrent-edit / post-reparse-fail / rollback-rename-fail.
- Test fixtures live under `testdata/writepath/` (synthetic format; no real Claude Code / Codex bytes).

**PRD/architecture refs.** PRD FR-5, NFR-T1; Architecture §4, §11.

**Test plan.**
- Happy: rows for plain rewrite, first-write against missing file, idempotent rewrite.
- Edge: every fault-injection row above, each asserting backup retention / rollback / exit-code semantics.

**No fallback writes.** Tests assert refuse-on-malformed; the synthetic adapter is the place to prove the invariant. NFR-S1.

**Complexity.** L.

**Dependencies.** E2-S2, E2-S3, E2-S4.
