# E6-S6 — `cmd/rename` + `cmd/delete` (`--yes`, clears active pointer)

**User story.**
As a claudecm user, I want `claudecm rename <old> <new>` and `claudecm delete <name>` to be safe, scriptable, and never leave the active-profile pointer dangling.

**Acceptance criteria.**
- Given `rename old new`, when run, then the profile file is renamed and validated against the profile-name regex; existing `new` refuses unless `--overwrite`.
- Given `delete <name>` interactively, when run, then it prompts before deleting; `--yes` skips the prompt.
- Given the deleted profile is the active profile, when delete completes, then the active-profile pointer is **cleared** in `state.yaml` (FR-2). It never silently points at a missing profile.
- Given `--dry-run`, when invoked, then the planned action is printed and nothing changes.
- Neither command modifies tool-config files. (Re-rendering happens on next `switch`.)

**PRD/architecture refs.** PRD FR-2, FR-15, NFR-S5; Architecture §2.2, §7.

**Test plan.**
- Happy: rename + delete + dry-run all behave as advertised.
- Edge: rename to invalid name → refused; delete active profile → active pointer cleared; `--yes` skips prompt.

**No fallback writes.** Bad name / missing profile / dangling pointer all refuse cleanly.

**Complexity.** S.

**Dependencies.** E1-S1, E1-S7.
