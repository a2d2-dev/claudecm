# E8-S1 — `testdata/` corpus scaffolding (NFR-T1 release gate)

**User story.**
As a claudecm maintainer, I want a pinned `testdata/` corpus laid out per NFR-T1 so each adapter's fixture matrix lives at a known path and the release gate is enforceable.

**Acceptance criteria.**
- Directory layout exists: `testdata/{claudecode,codex}/{happy,edge}/<case>/`.
- A `README.md` (one file, scoped to `testdata/`) documents the corpus contract: file names, expected golden outputs, how to add a fixture (paired PRD edit required for any owned-key change).
- Each case includes the source input(s), expected `Plan` output (golden), expected `Project` output (golden), and a per-case `assertions.yaml` describing edge characteristics (BOM, CRLF, etc.).
- This story is the *gate* (tracking): the actual fixture files land incrementally via E3-S7 and E4-S7.

**PRD/architecture refs.** PRD NFR-T1; Architecture §11.

**Test plan.**
- Happy: layout exists, README present.
- Edge: linter checks that any new `testdata/<tool>/edge/<case>/` includes the four required files; CI fails otherwise.

**No fallback writes.** N/A — fixtures only.

**Complexity.** S.

**Dependencies.** None.
