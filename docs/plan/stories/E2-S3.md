# E2-S3 — Post-write reparse + auto-rollback from FR-5 step-6 backup

**User story.**
As a claudecm developer, I want `writepath.Apply` step 8 to re-read the just-written file, parse it, and verify every owned key matches intent — and on any failure, auto-rollback from the FR-5 step-6 backup so the user never sees a half-written or drifted tool config.

**Acceptance criteria.**
- Given a successful step-7 rename, when step 8 runs, then the file is read back, parsed with the format-aware parser, and every owned-key value is checked against `WritePlan.Intended`. *(Owned-key drift bullet superseded — see Implementation Notes / Decision.)*
- Given the reparse fails OR any owned key drifted, when auto-rollback fires, then the FR-5 step-6 backup is renamed back over the target, `ApplyReport.RolledBack = true`, and an error names the failing key + backup path.
- Given the rollback rename itself fails (rare, e.g. filesystem read-only after write), when `Apply` returns, then the error is a structured `ErrRollbackFailed` carrying both the original failure and the rollback failure; exit code is 3.
- After auto-rollback, the original file is byte-identical to its pre-write content.

**PRD/architecture refs.** PRD FR-5 step 8; Architecture §4 step 8; coding-standards rule 7.

**Test plan.**
- Happy: write a valid plan → reparse passes, no rollback.
- Edge: inject a parser failure on reread → rollback fires and restores byte-identical original.

## Implementation Notes / Decision

Reparse validates parseability only; owned-key semantic drift is out of scope for E2-S3. Adapters are responsible for producing bytes whose semantic content matches their intent. Reparse catches: corrupted serialization, unclosed quotes, invalid TOML/JSON, encoding mismatches.

Rollback restores from in-memory pre-write bytes (captured under the flock before the write). The backup file is retained on disk and referenced in WriteReport.Backup for audit / user-invoked `claudecm restore`. This avoids a rename-across-devices failure mode and keeps rollback and audit independent.

**No fallback writes.** Post-write reparse failure does not "best-effort fix" the file; it rolls back. NFR-S1 / FR-5.
