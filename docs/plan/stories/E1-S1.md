# E1-S1 — Unified Profile schema with `schema_version: 1`

**User story.**
As a claudecm developer, I want the on-disk Profile model to match the locked unified schema (core fields + sparse `tools` overlay + `schema_version: 1`) so that every command above me reads and writes one consistent shape and rejects malformed or unknown-version profiles.

**Acceptance criteria.**
- Given a profile YAML with `schema_version: 1`, when `internal/config` reads it, then it returns a `Profile` with `Core` and an optional sparse `Tools` map.
- Given a profile YAML with `schema_version` missing, when it is read, then `internal/config` returns a clear error and never returns a partially-populated profile.
- Given a profile YAML with `schema_version >= 2`, when it is read, then `internal/config` refuses with a "newer claudecm wrote this" error and never silently misreads.
- Given a new profile written by `internal/config`, then `schema_version: 1` is always present and the file is mode `0600` inside a `0700` directory.
- The legacy fields (`auth_token`, `custom_env`) from the current `internal/config/profile.go` are migrated to the new shape; nothing references the legacy field names after this story.

**PRD/architecture refs.** PRD §4.1 FR-1, NFR-M1, NFR-S4; Architecture §2.1.

**Test plan.**
- Happy: round-trip a canonical profile (core + sparse `claude_code` overlay + sparse `codex` overlay) through write → read → equality.
- Edge: read a fixture with `schema_version` missing → error; read a fixture with `schema_version: 2` → refuse; read a fixture with both `core` and a tool overlay that sets only one field → fall-through resolution returns core value for the unset overlay key.

**No fallback writes.** A YAML with bad schema is **refused**; no rewrite, no migration, no "best effort". NFR-S1 / NFR-M1.

**Complexity.** M.

**Dependencies.** None.
