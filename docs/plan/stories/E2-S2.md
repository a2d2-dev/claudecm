# E2-S2 — `writepath.Apply` core pipeline (lock → read → parse → resolve symlink → diff → backup → atomic rename)

**User story.**
As a claudecm developer, I want `writepath.Apply` to implement FR-5 steps 1–7 as the only path to disk for a tool-owned file, so every command above me writes through one place and no caller can bypass the invariant.

**Acceptance criteria.**
- Given a `WritePlan`, when `Apply` runs, then it executes (1) `flock(LOCK_EX)`, (2) read + record `(size, mtime, sha256)`, (3) format-aware parse (TOML doc-model / sjson for JSON), (4) resolve symlink, (5) compute owned-key diff vs intended, (6) backup-with-verification, (7) atomic temp+rename — in order.
- Given an unparseable target, when step 3 runs, then `Apply` returns a clear error, **no backup is written**, **no write is attempted**. NFR-S1.
- Given a symlink that resolves outside `$HOME`, when step 4 runs, then `Apply` refuses with the resolved path in the error. NFR-S2.
- Given a successful `Apply`, when it returns, then `ApplyReport.BackupPath` points at the verified backup and `ApplyReport.Post.SHA256` is the hash of the bytes just written.
- `Apply` is the only function in the codebase that calls `os.Rename` against a tool-owned file path; CI lint rule enforces this.

**PRD/architecture refs.** PRD FR-5 steps 1–7, NFR-S1, NFR-S2; Architecture §4 steps 1–7; coding-standards rule 1, 2, 5.

**Test plan.**
- Happy: write through a synthetic adapter with all owned keys present → backup present, rename succeeded, mode `0600`.
- Edge: malformed JSON target → refuse, no backup; symlink at `~/.claude/settings.json` → `/etc/passwd` → refuse with resolved path.

**No fallback writes.** Parse failure refuses outright. NFR-S1.

**Complexity.** L.

**Dependencies.** E1-S2, E1-S3, E1-S4, E1-S6, E2-S1.
