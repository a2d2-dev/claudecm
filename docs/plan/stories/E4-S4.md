# E4-S4 — `codex`: `Plan` + render for both files

**User story.**
As a claudecm developer, I want `codex.Plan(profile)` to emit two `WritePlan`s (one per owned file) whose intended bytes update only the owned keys, so the commit package (E7) can stage and order them.

**Acceptance criteria.**
- Given a profile + current `config.toml` + current `auth.json`, when `Plan` runs, then it returns two `WritePlan`s with intended bytes computed via the TOML doc model (E4-S2) and surgical JSON edits to `auth.json`.
- Given the profile omits an owned key, when `Plan` runs, then overlay-as-truth (NFR-S6) removes the key from the intended bytes.
- Given comment/order preservation is impossible for an owned TOML key, when `Plan` runs, then `WritePlan.Warnings` includes the NFR-S7 warning.
- Plans are pure: no disk writes, no syscalls beyond reading the current targets.
- The owned-key diffs power `--dry-run` and `switch`'s pre-apply diff.

**PRD/architecture refs.** PRD FR-4, FR-5, NFR-S6, NFR-S7; Architecture §3, §10.

**Test plan.**
- Happy: typical profile + on-disk Codex config → two plans, both byte-identical on non-owned scope.
- Edge: profile omits `model` → owned key removed; provider `relay-a` not present in disk file → plan adds the section without disturbing other providers.

**No fallback writes.** Type mismatch / parser limitation → refuse + warn. NFR-S1 / NFR-S7.

**Complexity.** L.

**Dependencies.** E4-S2, E4-S3.
